# 要件定義書: タスク管理アプリ

作成日: 2026-04-03  
ステータス: 画面設計完了

---

## 1. プロジェクト概要

### 目的（Why）

React学習用として作成し、フロントエンド実装力と設計力を高める。  
コードの品質と保守性に重きを置き、実践的な開発プロセスを学ぶ。

### 概要（What）

タスクの詳細・実績時間を管理するWebアプリケーション。  
個人利用を前提としたローカルツールとして設計する。

---

## 2. 機能要件

### タスク管理

- タスクのCRUD（作成・読み取り・更新・削除）
- タスクの階層構造（深さ: 最大5階層 / 子タスク: 1ノードあたり最大20件）
  - **上限を設ける理由**: 無制限にするとツリーの再帰的なレンダリングが必要になり、UIの折りたたみ・インデント・スクロール制御が複雑化するため。仕様として上限を設けることで実装をシンプルに保つ。
- ステータス管理（未着手・進行中・完了 等）
- 並び替え（ドラッグ&ドロップ、または手動順序変更）

### 時間管理

- 開始 / 終了タイマーで実績時間を自動算出
- 実績時間を親タスクへ自動集計

### エクスポート

- タスク情報と実績時間を含むアプリ状態をJSON形式でエクスポート
- エクスポートしたJSONをインポートし、同じ状態を復元できる
- AIへの入力として渡すことを想定した設計
- **JSON形式に変更した理由**: タスクの階層構造と実績データをまとめて扱うため、表形式のCSVよりJSONの方がデータ構造を保持しやすい。実装規模も抑えられ、React学習用アプリとして優先度の高い状態管理・入力検証・インポート/エクスポート処理に集中できるため。

---

## 3. 非機能要件

### 対象ユーザー

- 自分のみ（認証機構なし）

### データ永続化

- **localStorage** を使用可能
  - デフォルトはOFF
  - `VITE_ENABLE_LOCAL_STORAGE_PERSIST=true` を設定した場合のみ有効
  - 有効化した場合、ブラウザを閉じても明示的に削除しない限り永続的に保持される
  - 容量の目安: 約5MB
- **デフォルトOFFに変更した理由**: GitHub Pagesで公開した際に、サイト訪問者のブラウザ環境へ意図せずデータを保存しないため。開発者が動作確認したい場合は環境変数で明示的に有効化する。

### エクスポート方式

- 手動ボタンによるダウンロード

## 4. 技術スタック

| 項目 | 選定内容 |
|------|---------|
| フレームワーク | React + Vite |
| 言語 | TypeScript |
| スタイリング | Tailwind CSS（予定） |
| テスト | Vitest + React Testing Library |
| データ永続化 | localStorage（環境変数で有効化） |
| デプロイ | GitHub Pages |

### Next.jsを採用しない理由

このアプリはページ遷移・SSR・API Routesを利用しない構成のため、Next.jsを導入する技術的なメリットがない。  

---

## 5. 不採用の決定事項

| 項目 | 理由 |
|------|------|
| File System Access API | ブラウザのセキュリティ制限により任意ディレクトリへの書き込みが不可（毎回ファイルピッカーによるユーザー操作が必要）。通常のダウンロード（手動エクスポートボタン）で要件を満たせるため不要と判断 |
| 認証機構 | 自分のみが利用するローカルツールのため不要 |
| Next.js | ページ遷移・SSR・API Routesの利用がなく技術的な必要性がない |

## 6. MVP定義

### Must（これがなければリリースできない）

- タスクのCRUD（作成・読み取り・更新・削除）
- タスクの階層構造（深さ最大5階層・子タスク最大20件）
- ステータス管理（未着手・進行中・完了 等）
- 開始/終了タイマーによる実績時間の算出
- データエクスポート（タスク情報 + 実績時間を含むJSON）
- データインポート
- localStorageによる永続化（環境変数で有効化）

### Should（あった方が良いが、なくても最低限使える）

- 実績時間の親タスクへの集計

### Could（時間があればやる）

- 並び替え（ドラッグ&ドロップ、または手動順序変更）

## 付録: 要件決定の経緯（Claude Codeとの会話サマリ）

この要件定義はClaude Codeとの対話を通じて決定した。主な意思決定の流れを以下に記録する。

### 要件決めのアプローチ

要件定義のセオリーに従い、以下の順序で整理した。

```
Why（目的・ゴール）
  → What（要件の洗い出し）
    → 機能要件 / 非機能要件
      → MVP定義（Must / Should / Could）
        → 技術スタック選定
```

### 主な意思決定の記録

**1. 対象ユーザー: 自分のみ（認証なし）**  
ローカルで利用するツールとして作成するため、認証機構は不要と判断。

**2. タスク階層の上限（深さ5・子タスク20件）**  
無制限にすると再帰的なレンダリングが必要になり、折りたたみ・インデント・スクロール制御が複雑化する。  
仕様として上限を設けることで実装をシンプルに保つ。

**3. データ永続化: localStorage（環境変数で有効化）**  
localStorageは明示的に削除しない限り永続的に保持される（ブラウザを閉じても消えない）。  
ただし、GitHub Pagesで公開した際にサイト訪問者のブラウザ環境へ意図せずデータを保存しないため、デフォルトはOFFとする。  
開発・動作確認時に `VITE_ENABLE_LOCAL_STORAGE_PERSIST=true` を設定した場合のみ有効化する。
File System Access APIはブラウザのセキュリティ制限により任意ディレクトリへの書き込みが不可（毎回ファイルピッカーによるユーザー操作が必要）なため見送り。  
通常の手動ダウンロード（エクスポートボタン）で要件を満たせると判断。

**4. フレームワーク: React + Vite / TypeScript（Next.jsは不採用）**  
ページ遷移・SSR・API Routesの利用がなく、Next.jsの技術的なメリットがない。  

**5. エクスポート形式: JSON**  
タスク情報と実績時間を含むアプリ状態をJSONで出力する。  
当初は実績CSV + タスクJSONを想定していたが、タスクの階層構造と実績データを一体で扱うにはJSONの方が適しているため、CSV出力はMVPから外す。  
React学習用アプリとしては、CSV整形よりも状態管理・入力検証・インポート/エクスポート処理の理解を優先する。
