# コードレビュー結果（2026-04-22）

対象ブランチ: `main`
観点: 設計 / 不具合 / セキュリティ

---

## 1. 設計

### 1-1. 責務分離は良好
- `useTaskContext`（state） / `useTimerContext`（timer UI 状態） / `useTaskManager`（タスク CRUD） / `useTimer`（計測） / `useTaskOperations`（タスク×タイマ整合）の層構造は明確。
- 特に `useTaskOperations` が「削除/import/clear 時にタイマをリセット」という横断的整合を一箇所に寄せている点は綺麗。
- zod の `superRefine` で import の整合（重複 ID、存在しない parent、循環、深さ、子数、時刻逆転）を同じスキーマに集めているのも読みやすい。

### 1-2. コンポーネント階層での `task` 取得の重複
`TaskNode` → `TaskRow` → `StatusBadge` / `TaskActionMenu` / `Timer` のいずれもが個別に `useTaskManager()` / `useTaskOperations()` を呼んで `getTask(taskId)` を実行している（`src/components/TaskTree/TaskNode.tsx:13`、`src/components/TaskTree/TaskRow/TaskRow.tsx:25`、`src/components/TaskTree/TaskRow/StatusBadge.tsx:12`）。

毎レンダーで以下が発生する:
- Context 購読 × コンポーネント数
- `tasks.find` の線形走査 × コンポーネント数

タスク数が増えるとレンダリングコストがタスク数の二乗で効く。親で取得した `task` を props で渡すか、`TaskContext` を selector 化する方が素直。

### 1-3. `useElapsed` のコストが重い
`useElapsed`（`src/hooks/useElapsed.ts:14`）は毎呼び出しで `getTaskDescendants` を再帰的に実行し、さらに各 ID について `trackRecords.filter(...).reduce(...)` を行う。結果として `O(タスク数 × 子孫数 × trackRecord数)` が各レンダーで走る。

ツリー全体を一度だけ集計して Map 化する形（`useMemo` で `Map<taskId, ms>` を作り、`TaskRow` から参照）の方が効率的。

### 1-4. `useTaskManager` / `useTaskOperations` の関数が毎レンダー再生成
どちらの hook も返り値の関数をメモ化していない（`useCallback` なし）。現在はすべて prop として葉に渡っているだけなので実害は少ないが、将来 `useEffect` の依存に入れたりメモ化コンポーネントに渡すと再実行のトリガになる。API が増えてきた今が整えどき。

### 1-5. `dispOrder` がデッドフィールド化
`addTask` では `dispOrder` を採番しているが（`src/hooks/useTaskManager.ts:56`）、`getChildren` / `getRootTasks` は `filter` のみで `dispOrder` ソートを行っていない（`src/utils/taskTree.ts:22-33`）。現状は配列への push 順に依存しており、採番した意味がない。

並び替え機能を「Could」としているなら、実装するまでフィールド自体を削除するか、少なくとも表示側でソートを噛ませるべき。

### 1-6. ルートタスクの上限チェックが抜けている【仕様確認済み: バグ確定】
`addTask` は `parentId` が存在する場合のみ `currentCount >= 20` を弾く（`src/hooks/useTaskManager.ts:49`）。`countRootTasks` を計算していながら使っていない。zod の `superRefine` もルートの件数を検証していない。

仕様として「ルートも 20 件制限」であることを確認済み。以下は明確なバグ:

- UI 上で「+ タスクを追加」を 21 回押すとすべて成功する
- 21 件以上のルートタスクを含む JSON を import しても通る

#### 修正範囲
1. **`src/hooks/useTaskManager.ts:49`**: ルート追加時のチェックを追加
   ```ts
   if (!params.parentId && currentCount >= 20)
     throw new Error("ルートタスクの上限を超えています");
   ```
   もしくは条件を統合して `currentCount >= 20` 単独でチェック（`currentCount` は parentId の有無で既に適切に計算済み）。

2. **`src/schemas/task.ts`** の `superRefine`: `getRootTasks(state.tasks).length > 20` の検証を追加。

#### テスト追加推奨
- `useTaskManager` のユニットテストに「ルート 20 件のときに 21 件目の追加が throw する」
- `TaskStateSchema` に「ルート 21 件の state が parse 失敗する」

### 1-7. `TaskRow` に `hasChildren` があるのに `Timer`/`StatusBadge` が再取得
`TaskRow` 側で `hasChildren` を props 経由で把握しているのに、`Timer` は内部で改めて `getTaskChildren(taskId).length > 0` をチェックしている（`src/components/TaskTree/TaskRow/Timer.tsx:12`）。props で渡すだけで済む。

## 2. 不具合・バグの懸念

### 2-1. 【高】localStorage に壊れた JSON があると起動時クラッシュ
`useLocalStorage` 初期化で `JSON.parse(item)` を try/catch なしで呼んでいる（`src/hooks/useLocalStorage.ts:18`）。拡張機能・開発中のスクリプト・ユーザー操作などで `"task-manager"` キーが壊れた値になると、アプリ全体が初期レンダーで例外を吐いて表示されなくなる。

`TaskProvider` 側には `TaskStateSchema.parse` の try/catch があるのに、その手前でコケる。`JSON.parse` も try/catch で包み、失敗時は `initialValue` に倒すべき。

### 2-2. 【中】編集モーダル表示中に対象タスクが消えるとクラッシュ
`src/hooks/useTaskFormModal.ts:62` の

```ts
const editingTask =
  modalState.isOpen && modalState.mode === "edit"
    ? taskOperations.getTask(modalState.taskId)
    : null;
```

は毎レンダーで `getTask` を呼び、未存在なら `throw new Error("対象タスクが存在しません")` する。

現状の UI フローでは「編集中に別の操作で削除」というパスは無さそうだが、import/clear が走って state が丸ごと差し替わると起こりえる（`importState` 内で `timer.reset` は呼んでいてもモーダルはリセットされていない）。`find` 相当の安全取得に変え、見つからなければモーダルを閉じる方が安全。

### 2-3. 【低】`FileReader` の失敗パスが無言で消える
`src/components/Header/Header.tsx:22` の `reader.onerror` が未定義で、`event.target?.result` が null/ArrayBuffer のケースも `as string` で強引に通している。

`readAsText` 前提なら通常は string だが、null のとき `JSON.parse(null as unknown as string)` → `"null"` 経由で `null` → zod が弾いて「不正なファイル形式です」と alert、という流れには乗る。しかし `onerror` 側の読み取りエラーは捕まらずログも出ない。最低限 `reader.onerror` を足して `handleAppError` を呼ぶのが良い。

### 2-4. 【高】タイマー動作中のリロードで計測が消える
`TimerContext` は `useState` のみで永続化されていない（`src/contexts/TimerContext.tsx:9-10`）。タブを閉じる・リロード・クラッシュで `runningTaskId` と `startedAt` が失われ、計測中の記録は `trackRecord` として保存されない（`stop` が呼ばれない）ため、数時間の作業が丸ごと失われる可能性がある。

`localStorage` を有効化している環境でも、`trackRecord` として確定するのは `stop` 時点なので同じ。要件に記載はないが、実用時の地雷。最低限「計測中のタスク ID と開始時刻だけ永続化し、復帰時に継続」くらいは考慮に値する。

### 2-5. 【要確認】`formatElapsed` の集計タイミング
`useTimer.stop()` では個別 `trackRecord` について `formatElapsed` の値が 0 より大きいものだけ保存している（`src/hooks/useTimer.ts:33`）。つまり「14 分の計測は 0h 扱いで保存自体されない」設計。

集計側 `useElapsed` は合算してから `formatElapsed` するので、14 分を何十回積んでも合計実績は 0h。要件としてそれが意図なら OK だが、通常の「0.25h 単位に切り捨てた合計」を期待する UX とはズレる。個別保存せず生のミリ秒で残し、表示側で切り捨てる方が一貫する。

### 2-6. 【軽微】`useTaskManager.deleteTask` は再帰で古い state を参照する
`state.trackRecords.filter(...)` と `getChildren(state.tasks, id)` は dispatch 直後も旧 state を見続ける（`src/hooks/useTaskManager.ts:94-102`）。削除対象は最終的に全て網羅されるため結果的に正しいが、`deleteTask` 内で複数回 dispatch を流す構造は可読性が低く、将来トランザクション境界を入れづらい。

削除対象 ID を一括で集めて 1 つの reducer アクション（例: `DELETE_TASK_SUBTREE`）に寄せる方が素直。

### 2-7. 【軽微】`clearState` のフラグベース副作用制御
`TaskProvider.clearState` は `removeStoredState` → `dispatch(IMPORT, initialState)` の順で、その間で一時的に `storedValue === initialValue` と `reducer state !== initialValue` が同時に存在する瞬間がある。React のバッチングで実害は出にくいが、`skipNextPersistRef` のような「次の effect をスキップ」パターンは `StrictMode` や将来の並行モードで壊れやすいフラグ。

`IMPORT` と `CLEAR` を別アクションにし、`CLEAR` 時にだけ `setStoredState` を呼ばず `removeStoredState` を呼ぶ、という形にすると副作用順序を気にせず済む。

## 3. セキュリティ

### 3-1. 【低】`localStorage` キーの同一オリジン衝突
#### 前提
localStorage は同一オリジンポリシーで保護されている。**別オリジンの第三者サイトから勝手に読み取られることはない**。

#### 懸念の範囲
GitHub Pages の `username.github.io/<repo>/` は、ブラウザから見ると **`username.github.io` 全体で 1 つのオリジン**。localStorage はパスではなくオリジン単位でスコープされるため、同じ `username.github.io` 配下に公開した別プロジェクトの JavaScript は、このアプリの localStorage を読める。

成立する条件:
1. **キー名の衝突で事故**: 別プロジェクトも偶然 `"task-manager"` というキーを使っていたら上書きされる（攻撃ではなく単純な事故）。
2. **別プロジェクトに XSS 脆弱性**: 同じ GitHub アカウントで公開している別のブログや遊び場ページに XSS があれば、そこから流れ込んだスクリプトが task-manager のデータを読める。
3. **ブラウザ拡張機能**: オリジン制限を越えるので別話。

#### 結論
- 外部の第三者サイトから直接抜かれる心配はない
- リスクは「同じ `username.github.io` 配下に公開する他のコンテンツが事故や脆弱性を持っていた場合の巻き込まれ」に限定される
- 他に公開物がない、または今後も置かない予定なら実害はほぼ無視できる

#### 対策（任意）
- キーを `task-manager:v1` などプロジェクト固有名にして事故衝突を避ける

### 3-2. 【低】インポート JSON のサイズ・深さに対する DoS 耐性なし
`FileReader.readAsText` はサイズ制限なし、zod バリデーションもフル走査。悪意ある（あるいは壊れた）巨大 JSON を読ませると UI が固まる。localStorage に 10MB 書き込めばクォータ超過でエラー。

自身利用が前提でも誤操作由来でハングするので、ファイルサイズ上限（例 5MB）とタスク/trackRecord 件数上限のチェックを入れると堅牢。

### 3-3. 【低】XSS は基本安全、ただし将来 `dangerouslySetInnerHTML` を入れないこと
現状 `task.taskName`・`description` は全て React の子として描画されており自動エスケープされる。import で任意文字列が混入しても XSS にはならない。将来 Markdown レンダ等を入れるならサニタイザ必須。

### 3-4. 【低】`window.alert` による情報出力
`handleAppError` が `alert` でエラーメッセージを表示する（`src/utils/error.ts:12`）。個人利用なら OK だが、将来スタックトレースや詳細情報を含めるときは alert に流さず UI のトースト/パネルへ流す想定で。ログ用途で `console.error(error)` も併用しておくとデバッグしやすい。

### 3-5. 【情報】`.env.local` 関連
`.gitignore` の `*.local` で除外されており、コミット履歴にも入っていない想定。`VITE_ENABLE_LOCAL_STORAGE_PERSIST` は秘密情報ではないので問題なし。
